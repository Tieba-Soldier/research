# 资源推荐功能接入国内搜索能力改造方案

## 1. 背景

当前资源推荐功能的搜索链路基本完全依赖 Tavily：

- `backend/app/tools/tavily_tool.py`
- `backend/app/services/resource_search_service.py`
- `backend/app/services/search_service.py`
- `backend/app/core/config.py`

这会带来两个直接结果：

1. 搜索结果天然偏英文站点和海外内容生态。
2. 项目里“搜索能力”被写死为 Tavily，后续想切到国内搜索源时，改动会穿透到多个 service。

你的目标不是简单“把 Tavily 换掉”，而是让项目具备 **国内资源优先** 的搜索能力，同时保留后续扩展为“国内 + 海外混合搜索”的空间。

## 2. 改造目标

建议把这次改造的目标定成下面四点：

1. 支持国内搜索 API，优先返回中文、国内平台、国内开发者生态里的资源。
2. 保持现有推荐流程不大改，搜索模块做可插拔化。
3. 保留 Tavily 作为 fallback，避免国内搜索服务波动时系统完全不可用。
4. 为后续“按地区/语言/资源类型选择搜索源”预留接口。

## 3. 现状问题

### 3.1 搜索能力被写死

目前 `ResourceSearchService` 和 `SearchService` 都直接 `new TavilyTool()`，不是通过抽象接口或工厂获取 provider。

这意味着：

- 新增一个国内搜索源时，需要在多个 service 里改构造逻辑。
- 无法按环境变量切换 provider。
- 很难做 `cn / global / hybrid` 三种模式。

### 3.2 配置项只支持 Tavily

`backend/app/core/config.py` 里只有：

- `TAVILY_API_KEY`
- `ENABLE_TAVILY`

没有统一的搜索 provider 配置，也没有 region、语言、fallback 策略配置。

### 3.3 缓存 key 不区分 provider

`backend/app/services/cache_service.py` 的搜索缓存目前只按 `query` 做 key。

如果未来同一个 query 既可能走 Tavily，也可能走国内搜索 API，那么缓存会串结果。

### 3.4 查询词仍然是“中性搜索词”

现在搜索 query 基本来自主题标准化后的 topic / keywords，没有做“中文资源导向”的补词。

例如：

- `Python asyncio`

对于国内资源，更合适的 query 往往应该变成：

- `Python asyncio 教程`
- `Python asyncio 中文 官方文档`
- `Python asyncio B站`
- `Python asyncio 实战`

## 4. 国内搜索 API 选型建议

## 4.1 推荐结论

截至 **2026-04-19**，我建议采用下面的策略：

### 首选：博查 Search API

适合作为你项目里的 **主国内搜索 provider**。

推荐原因：

1. 它本身就是面向 AI / Agent / RAG 场景的搜索 API，返回结构化结果，接入方式和 Tavily 很像。
2. 官方文档明确提供 Web Search API，支持 `query`、`count`、`freshness`、`summary` 等参数。
3. 官方页面展示了中文网页、新闻、百科、视频等内容场景，更适合“中文学习资源推荐”。
4. 它是独立搜索 API，不需要先绑定某个大模型平台，工程接入最轻。

适配度判断：**高**

### 备选：智谱 Web Search API

适合作为 **第二选择** 或者 **国内备用 provider**。

推荐原因：

1. 官方文档明确说明其联网搜索能力整合了 **智谱自研引擎 + 搜狗/夸克**。
2. 对中文互联网的覆盖天然比 Tavily 更友好。
3. 提供结构化搜索结果，也支持域名过滤、时间过滤、摘要长度控制。
4. 如果你后续 LLM 侧也偏国内生态，和智谱体系联动会更顺。

适配度判断：**中高**

### 不建议作为第一阶段主方案：直接接百度 / 搜狗 / 360 官方搜索结果 API

原因不是“它们不好”，而是 **截至 2026-04-19，我没能稳定验证到面对普通开发者、文档清晰、可直接拿来做通用网页搜索结果拉取的官方公开方案**。

更准确地说：

- 搜狗公开平台可以看到站长/数据接入/服务接入能力，但这和“通用 SERP 搜索结果 API”不是一回事。
- 360 官方公开搜索结果 API 文档没有查到稳定入口。
- 百度侧虽然公开信息里能看到“搜索 API”相关表述，但面向普通开发者直接落地的公开文档入口、权限方式、接入流程稳定性，目前不如博查/智谱清晰。

所以第一阶段不建议把项目绑在“可能需要商务接入或灰度资格”的搜索能力上。

## 4.2 选型对比

| 方案 | 类型 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| 博查 Search API | 独立搜索 API | 接入轻、结构化结果友好、中文资源适配强 | 需要新接入供应商 | **首选** |
| 智谱 Web Search API | 搜索 + 大模型平台能力 | 中文互联网覆盖好，支持搜狗/夸克 | 生态耦合更强，配置略重 | **备选 / fallback** |
| Tavily | 海外 AI Search API | 现有已接入、稳定 | 国内中文资源命中一般 | **保留 fallback** |
| 百度/搜狗/360 官方直连 | 理论上最本土 | 公开文档和可直接接入性不够稳定 | 接入风险高 | **暂不作为一期方案** |

## 5. 推荐的项目改造方式

不要写成“国内搜索版 TavilyTool”。正确做法是把搜索层做成 **provider 抽象 + 工厂 + 统一返回结构**。

## 5.1 目标结构

建议改成下面这种结构：

```text
backend/app/tools/
  search_provider.py          # 抽象接口
  tavily_tool.py              # 现有 provider，按接口适配
  bocha_tool.py               # 新增：博查 provider
  zhipu_search_tool.py        # 可选：智谱 provider
  search_provider_factory.py  # 按配置返回 provider
```

也可以再细分到 `backend/app/tools/search/` 目录，但如果你想控制改动面，一期直接放在 `tools` 下就够了。

## 5.2 统一返回结构

无论底层接的是 Tavily、博查还是智谱，service 层拿到的都应该是同一个 schema。

建议统一为：

```python
{
    "title": str,
    "url": str,
    "snippet": str,
    "score": float,
    "source": str,
    "provider": str,
    "language": str | None,
    "published_at": str | None,
    "raw": dict,
}
```

说明：

- `provider`：记录结果来自 `tavily` / `bocha` / `zhipu`
- `language`：后续做中文优先排序很有用
- `published_at`：新闻、教程更新时效性很有用
- `raw`：保留原始返回，便于排障和后续能力扩展

## 5.3 搜索接口抽象

建议定义一个统一接口：

```python
from typing import Protocol, List, Dict, Any


class SearchProvider(Protocol):
    async def search(
        self,
        query: str,
        max_results: int = 5,
        timeout: int = 30,
    ) -> List[Dict[str, Any]]:
        ...
```

然后：

- `TavilyTool` 实现这个接口
- `BochaTool` 实现这个接口
- `ZhipuSearchTool` 实现这个接口

这样 service 层就不用知道底层是谁。

## 5.4 工厂和配置切换

建议新增配置：

```env
SEARCH_PROVIDER=bocha
SEARCH_REGION=cn
SEARCH_FALLBACK_PROVIDER=tavily

BOCHA_API_KEY=your_bocha_api_key
ENABLE_BOCHA=true

ZHIPU_API_KEY=your_zhipu_api_key
ENABLE_ZHIPU_SEARCH=false
```

并在 `Settings` 中新增：

- `SEARCH_PROVIDER: str = "tavily"`
- `SEARCH_REGION: str = "global"` 取值建议：`global | cn | hybrid`
- `SEARCH_FALLBACK_PROVIDER: Optional[str] = None`
- `BOCHA_API_KEY: Optional[str] = None`
- `ENABLE_BOCHA: bool = False`
- `ZHIPU_API_KEY: Optional[str] = None`
- `ENABLE_ZHIPU_SEARCH: bool = False`

工厂逻辑建议：

1. 先根据 `SEARCH_PROVIDER` 返回主 provider。
2. 如果主 provider 异常，再看 `SEARCH_FALLBACK_PROVIDER` 是否可用。
3. service 层只关心调用 `provider.search()`。

## 6. 需要改哪些代码

## 6.1 `backend/app/core/config.py`

这里是第一处必须改的地方。

需要做的事：

1. 保留 Tavily 相关配置。
2. 新增国内搜索 provider 配置。
3. 加入统一搜索策略配置。

建议新增：

```python
# Unified Search
SEARCH_PROVIDER: str = "tavily"
SEARCH_REGION: str = "global"
SEARCH_FALLBACK_PROVIDER: Optional[str] = None

# Bocha
BOCHA_API_KEY: Optional[str] = None
ENABLE_BOCHA: bool = False

# Zhipu Web Search
ZHIPU_API_KEY: Optional[str] = None
ENABLE_ZHIPU_SEARCH: bool = False
```

如果你准备做“中外混合”，还可以再加：

- `CN_SEARCH_DOMAIN_BOOSTS`
- `GLOBAL_SEARCH_DOMAIN_BOOSTS`
- `SEARCH_TIMEOUT_SECONDS`

## 6.2 `backend/app/tools/tavily_tool.py`

不要删，保留为 legacy/global provider。

建议改动：

1. 让它遵循统一的搜索接口。
2. 返回统一 schema。
3. 给结果补上 `provider="tavily"`。

## 6.3 新增 `backend/app/tools/bocha_tool.py`

这是本次改造的核心文件。

职责：

1. 调博查 Web Search API。
2. 把结果映射成统一 schema。
3. 遇到超时或 API 错误时抛出 `SearchException`。

建议实现方式：

- 用 `httpx.AsyncClient` 调 REST API。
- 保持和现有 `TavilyTool.search()` 一样的 async 风格。
- 统一做 timeout、错误处理、日志记录。

建议映射：

- `title` <- 博查结果 `name`
- `url` <- 结果 `url`
- `snippet` <- `summary` 优先，其次 `snippet`
- `source` <- `siteName` 或域名
- `score` <- 如果没有明确评分，先给默认值，如 `0.0`
- `provider` <- `"bocha"`

## 6.4 可选新增 `backend/app/tools/zhipu_search_tool.py`

如果你希望一期就有国内 fallback，可以一起做。

职责和 `BochaTool` 一样，只是底层接智谱 Web Search API。

建议定位：

- 不是默认主 provider
- 作为 `SEARCH_FALLBACK_PROVIDER=zhipu`
- 或者用于 `SEARCH_REGION=cn` 时的第二搜索源

## 6.5 新增 `backend/app/tools/search_provider_factory.py`

建议把 provider 选择集中在这里，不要散落在 service 层。

工厂返回逻辑示例：

```python
if settings.SEARCH_PROVIDER == "bocha":
    return BochaTool()
if settings.SEARCH_PROVIDER == "zhipu":
    return ZhipuSearchTool()
return TavilyTool()
```

后续也可以扩展成：

- `get_primary_provider()`
- `get_fallback_provider()`
- `get_hybrid_providers()`

## 6.6 `backend/app/services/resource_search_service.py`

现在这里直接依赖 `TavilyTool()`，建议改为依赖工厂。

要改的点：

1. `__init__` 中改成 `self.search_provider = get_search_provider()`
2. 所有 `self.tavily_tool.search(...)` 改为 `self.search_provider.search(...)`
3. 保存结果时把 `provider` 一并保留下来

例如构造资源时增加：

```python
"search_provider": result.get("provider", settings.SEARCH_PROVIDER)
```

虽然数据库里现在没有这个字段，但至少内存对象里保留，便于后续扩展。

## 6.7 `backend/app/services/search_service.py`

这个文件比 `resource_search_service.py` 更关键，因为它是 v2 推荐链路里的主搜索入口。

建议改动：

1. 构造函数改成通过工厂拿 provider。
2. 缓存逻辑加入 provider 维度。
3. 允许 `cn / global / hybrid` 三种搜索模式。

### 最小改法

只切换 provider，不改整体流程：

- `self.tavily_tool` -> `self.search_provider`
- `query` 缓存 key 加 provider

### 更好的改法

增加模式判断：

- `global`：走 Tavily
- `cn`：走 Bocha
- `hybrid`：Bocha + Tavily 并发搜索，合并去重后返回

我更推荐你这样做，因为这会把“国内优先”从一次性 hack 变成长期能力。

## 6.8 `backend/app/services/cache_service.py`

搜索缓存一定要改。

当前：

```python
key = self._generate_key("search", query)
```

建议改成：

```python
cache_key = f"{provider}|{region}|{query}"
key = self._generate_key("search", cache_key)
```

否则同一个 `query` 在 Tavily 和博查之间会相互污染缓存。

## 6.9 `backend/app/services/recommendation_service_v2.py`

这个文件不用大改，但最好补两件事：

1. 指标日志不要再写死 `tavily_calls`
2. 改成更通用的 `search_provider_calls`

现在的日志会误导后续运维判断。

建议改为：

- `search_provider_calls`
- `search_provider_name`

## 6.10 数据库存储层（可选）

当前 `resources` 表已经有 `source` 字段，但这个字段更多像“站点来源”，不是“搜索来源”。

如果你后面希望追踪“这个资源是 Tavily 找到的还是博查找到的”，建议新增字段：

- `search_provider`

这不是一期必须项，但非常建议做。

## 7. 查询策略要一起改

只换 API 不够，query 也要一起“中文化”。

## 7.1 当前问题

当前 query 基本是：

- 标准化主题
- 若干关键词

这更适合通用搜索，不足以把结果强行拉向中文学习资源。

## 7.2 建议新增中文资源搜索模板

对于每个主题，除了原始 query，再扩展 3 到 5 个中文导向 query。

例如主题是 `Python asyncio`，建议生成：

1. `Python asyncio 教程`
2. `Python asyncio 中文 官方文档`
3. `Python asyncio 实战`
4. `Python asyncio B站`
5. `Python asyncio 掘金 OR CSDN`

对于 AI/编程学习资源，建议优先命中的国内站点：

- `bilibili.com`
- `juejin.cn`
- `csdn.net`
- `zhihu.com`
- `cloud.tencent.com`
- `developer.aliyun.com`
- `huaweicloud.com`
- `volcengine.com`
- `gitee.com`

注意不要把域名白名单写得太死，否则会错过优质中文独立博客。

更合理的策略是：

- 白名单做 **boost**
- 不是 **hard filter**

## 7.3 资源类型识别也要补国内站点

现在 `_infer_resource_type()` 已经识别了 `bilibili.com`，这很好，但还不够。

建议增加：

- `gitee.com` -> `code_repo`
- `zhihu.com` -> `article`
- `juejin.cn` -> `article`
- `cloud.tencent.com` / `developer.aliyun.com` / `huaweicloud.com` -> `official_doc` 或 `article`

## 8. 推荐实施路径

## 8.1 一期：最小可用改造

目标：一周内落地“国内搜索可切换”。

做法：

1. 新增 `BochaTool`
2. 新增 `SearchProvider` 抽象和工厂
3. 修改 `SearchService` / `ResourceSearchService` 改走统一 provider
4. 修改缓存 key，加入 provider
5. 新增配置项，默认 `SEARCH_PROVIDER=bocha`
6. 保留 Tavily 作为 fallback

一期完成后，你就能做到：

- 国内模式：`bocha`
- 海外模式：`tavily`
- 通过环境变量切换

## 8.2 二期：混合搜索

目标：中文资源优先，但必要时补全球结果。

做法：

1. 增加 `SEARCH_REGION=hybrid`
2. `Bocha + Tavily` 并发搜索
3. 合并结果后统一去重
4. 用规则做 rerank

建议 rerank 规则：

- 中文站点 +2
- 国内开发者社区 +2
- 官方文档/云厂商文档 +3
- B 站优质教程 +2
- 近一年内容 +1
- 海外英文资源不降为 0，但优先级略低

这样更符合“国内资源优先，而不是国内资源唯一”的产品目标。

## 8.3 三期：垂直站点增强

如果后续你发现“通用搜索 API 找到的中文技术资源还是不够稳”，再继续做：

1. B 站搜索专用接入
2. Gitee 搜索专用接入
3. 国内技术社区内容源聚合
4. 站点权重配置化

这一阶段才值得考虑更重的定制。

## 9. 推荐的最终架构

建议你最终把搜索能力做成下面这个结构：

```text
RecommendationServiceV2
  -> SearchService
      -> SearchProviderFactory
          -> BochaTool (primary for cn)
          -> TavilyTool (fallback/global)
          -> ZhipuSearchTool (optional backup)
      -> QueryBuilder
      -> CacheService
      -> ResultMerger / Deduplicator / Reranker
```

这样后续不管你是：

- 切换 provider
- 增加混合搜索
- 增加按资源类型搜索
- 增加站点权重

都不需要再把 service 层拆一遍。

## 10. 我给你的明确建议

如果你现在就要做，我建议按下面这套执行：

### 方案

- 主搜索源：**博查 Search API**
- 备用搜索源：**Tavily**
- 可选第二备用：**智谱 Web Search API**

### 搜索模式

- 默认：`SEARCH_REGION=cn`
- provider：`SEARCH_PROVIDER=bocha`
- fallback：`SEARCH_FALLBACK_PROVIDER=tavily`

### 代码策略

- 不直接在 service 里写死 Tavily
- 先抽象 provider，再接 Bocha
- 缓存 key 带 provider
- query 增加中文资源导向模板

### 为什么这样最合适

因为这套方案兼顾了：

1. 落地速度
2. 国内结果质量
3. 风险可控
4. 后续扩展性

## 11. 官方资料与依据

以下内容是我在 **2026-04-19** 查到并据此做出的选型判断：

1. [博查 AI 开放平台首页](https://open.bochaai.com/)
   - 官方页面明确展示 `Web Search API`
   - 明确给出 `https://api.bochaai.com/v1/web-search`
   - 明确展示中文网页搜索结果和结构化响应

2. [博查开放平台总览](https://open.bochaai.com/overview)
   - 官方页面明确列出 `Web Search API`
   - 页面显示可领取 `1000` 次调用资源包

3. [智谱联网搜索官方文档](https://docs.bigmodel.cn/cn/guide/tools/web-search)
   - 官方文档明确说明整合 **智谱自研引擎及第三方服务（搜狗/夸克）**
   - 提供结构化 Web Search API
   - 支持域名过滤、时间过滤、摘要长度控制

4. [搜狗开放平台首页](https://data.open.sogou.com/)
   - 能看到开放平台能力，但更偏数据接入、站长和开放展示
   - 不适合作为你项目一期的通用搜索结果 API 直接依赖

## 12. 一句话结论

这次不要做“把 Tavily 改成另一个 API”，而要做成：

**把搜索层抽象成可插拔 provider，然后用博查作为国内主搜索源，Tavily 作为全球 fallback。**

这才是既能解决“国内资源不够”的问题，又不会把项目下一次改造成本继续抬高的方案。
