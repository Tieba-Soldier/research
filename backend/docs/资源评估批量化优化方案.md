# 资源评估批量化优化方案

## 1. 背景

当前推荐链路中的资源评估阶段，对外接口看起来支持批量资源输入，但实际执行路径仍然是“逐条资源单独调用 LLM，再通过 `asyncio.gather()` 并发”。

这意味着：

- 资源数 = LLM 请求数
- 每次请求都会重复携带同一份 prompt 和同一份 `user_context`
- 当资源数增长时，输入 token 与请求数近似线性增长
- 当前架构更像“并发单条”，而不是真正的“批量评估”

在模型按上下文分档计费的前提下，资源评估阶段已经成为推荐链路中最明显的 token 消耗点之一，因此有必要对其进行批量化改造。

---

## 2. 当前实现现状

### 2.1 服务层表面是批量，内部是逐条任务

`ResourceEvaluationService.evaluate_resources()` 接收的是资源列表，但内部直接拆成逐条任务：

- 文件：`backend/app/services/resource_evaluation_service.py`
- 关键逻辑：对每个 `resource` 调一次 `_evaluate_single_resource(resource, user_context)`，再 `asyncio.gather()`

因此当前链路已经决定：**资源数量等于评估请求数量**。

### 2.2 节点层签名只支持单资源

`ResourceEvaluator.evaluate()` 的签名为单资源输入：

- 文件：`backend/app/agents/nodes/resource_evaluator.py`
- 关键签名：`async def evaluate(self, resource: Dict[str, Any], user_context: str, task_id: int = None)`

虽然 prompt 中使用了数组占位，但代码传入的是：

```python
json.dumps([resource], ensure_ascii=False, indent=2)
```

因此这里的“数组”只是将单条资源包成长度为 1 的列表，属于提示词适配层的伪批量。

### 2.3 归一化逻辑只关心第一条结果

当前归一化逻辑中存在如下处理：

```python
evaluated = evaluated_resources[0] if evaluated_resources else {}
```

这说明节点层的真实预期仍然是“一次请求只返回一条资源的评估结果”。

### 2.4 实际生效的 prompt 路径

当前 `ResourceEvaluator` 实际加载的 prompt 为：

- `backend/app/agents/prompts/resource_evaluator.md`

不是另一个同名的 `backend/app/prompts/resource_evaluator.md`。后者更像历史遗留文件，不应作为本次改造的依据。

---

## 3. 核心问题

### 3.1 重复固定开销过大

当前每次资源评估请求都会重复发送以下内容：

- 完整的评估 prompt
- 相同的 `user_context`
- 相同的输出格式约束
- 单条资源的 JSON 包装

资源本身并不算重，但固定提示词较长，因此在逐条调用模式下，固定成本被重复放大。

### 3.2 token 消耗与资源数近似线性增长

当前可近似表示为：

```text
总输入 token ≈ 资源数 * (固定 prompt 开销 + 单资源输入开销)
```

这会导致：

- 资源越多，请求数越多
- 请求总 token 增长较快
- 网络开销、超时概率、日志噪音也随之上升

### 3.3 当前 LLM 实际只负责部分字段

从归一化逻辑看，当前真正依赖 LLM 的主要是：

- `summary`
- `reason`
- `quality_score`
- `difficulty_level`

而以下字段并非由 LLM 决定：

- `relevance_score`：由 `raw_score` 推导
- `practical_score`：由 `resource_type` 推导
- `final_score`：由本地加权公式计算

这意味着当前调用中存在“模型调用成本较高，但模型实际负责字段较少”的情况，更适合做批量化压缩重复开销。

---

## 4. 优化目标

本次优化目标如下：

1. 将资源评估从“并发单条”改为“真批量输入”
2. 在不显著降低评估质量的前提下，减少总输入 token
3. 将批次控制在价格低档位内，避免因单次上下文过大导致单价跃升
4. 保持现有缓存、降级、超时保护能力
5. 尽量控制改造范围，优先兼容现有调用链路

非目标：

- 不改造搜索阶段
- 不改造学习路径和练习题生成阶段
- 不引入新的模型厂商或新的推理框架
- 不在本阶段做复杂的跨请求重排序策略

---

## 5. 成本模型与收益预估

## 5.1 粗略成本模型

基于当前代码结构，可将单次请求拆为两部分：

- 固定开销 `F`：prompt + `user_context` + JSON 外壳
- 单资源开销 `R`：单条资源的标题、URL、摘要、类型、原始分数等

近似模型：

```text
当前逐条模式：总输入 token ≈ N * (F + R)
批量模式：总输入 token ≈ ceil(N / B) * F + N * R
```

其中：

- `N` 为资源总数
- `B` 为 batch size

优化收益主要来自：**将固定开销从重复 N 次，降低为重复 ceil(N / B) 次**。

## 5.2 为什么不能无脑做超大批量

模型输入价格按上下文分档：

- `<= 32k`
- `32k ~ 128k`
- `128k ~ 256k`

因此批量化虽然通常能减少总 token，但如果单次请求跨过 `32k`，可能导致：

- 单价提高
- 单次失败影响范围扩大
- 解析结果错位风险上升
- 超时和截断概率增加

因此最优方案通常不是“所有资源一次送完”，而是“在低档位内做小批量或中等批量”。

## 5.3 推荐批量策略

结合当前资源字段体积和价格分档，推荐：

- 默认采用 **micro-batch**
- 默认批大小建议：`20 ~ 40`
- 以“单批输入不超过 28k token”作为软预算
- 留出 3k ~ 4k token 的安全余量，避免因为用户上下文变长、资源摘要偏长、模型消息包装额外开销而踩线

---

## 6. 方案选型

## 6.1 方案 A：保持逐条，只优化 prompt

做法：

- 精简 `resource_evaluator.md`
- 缩短输出字段要求
- 缩短 `summary` / `reason`

优点：

- 改动小
- 风险低
- 可快速见效

缺点：

- 请求数不变
- 固定开销仍按资源数重复
- 收益有限

适用场景：

- 需要超低风险止血
- 不希望改动服务层接口

## 6.2 方案 B：全量批量，一次评估全部资源

做法：

- 所有 cache miss 资源一次性送入 LLM

优点：

- 请求数最低
- 固定开销重复次数最少

缺点：

- 容易跨 32k 档位
- 失败影响面大
- 超时、截断、解析错位风险更高
- 资源数量波动时不稳定

适用场景：

- 资源数量稳定且规模很小

## 6.3 方案 C：动态 micro-batch 批量评估

做法：

- 服务层先查缓存
- 仅对 cache miss 的资源进行分批
- 按资源体积估算 token
- 在预算阈值内动态组装批次
- 批次间并发，批次内单次调用

优点：

- 收益明显
- 易于控制价格档位
- 失败影响范围可控
- 更适合现有架构渐进改造

缺点：

- 需要增加批次组装与结果映射逻辑
- 实现复杂度高于方案 A

**推荐结论：采用方案 C。**

---

## 7. 推荐目标架构

## 7.1 服务层职责调整

`ResourceEvaluationService.evaluate_resources()` 调整为：

1. 遍历输入资源，按 URL 查询评估缓存
2. cache hit 直接复用
3. cache miss 进入待评估列表
4. 对待评估列表按 token 预算分批
5. 并发执行批量评估
6. 将批量结果与缓存结果合并，恢复原始资源顺序

目标是让服务层成为“缓存 + 分批 + 聚合”的总入口。

## 7.2 节点层改为真批量输入

`ResourceEvaluator.evaluate()` 应升级为：

```python
async def evaluate(
    self,
    resources: List[Dict[str, Any]],
    user_context: str,
    task_id: int = None
) -> List[Dict[str, Any]]:
```

节点层应一次接收一个批次的资源列表，直接把整个 `resources` 数组送入 prompt，而不是再包装单个 `[resource]`。

## 7.3 结果映射与归一化

批量调用后不能再依赖：

```python
evaluated_resources[0]
```

应改为：

1. 将 LLM 返回的 `resources[]` 转换为映射表
2. 优先使用稳定键匹配，例如 `url`
3. 若 `url` 缺失或被模型改写，则回退到输入顺序匹配
4. 对每条输入资源执行统一归一化
5. 对缺失项执行单条降级或默认值补齐

推荐匹配优先级：

```text
url -> title+url 模糊匹配 -> 输入顺序回退
```

## 7.4 prompt 设计调整

prompt 仍保留资源数组输入，但应强化以下约束：

- 输出 `resources` 数组长度必须与输入一致
- 不得合并资源
- 不得遗漏资源
- 保留原始 `url`
- 输出顺序与输入顺序一致

建议在 prompt 中加入明确规则：

```text
请逐条评估输入数组中的每个资源。
输出 resources 数组长度必须与输入一致。
输出顺序必须与输入顺序一致。
每个输出项必须保留对应输入资源的原始 url。
```

这会显著降低批量模式下的错位成本。

## 8. 详细改造点

## 8.1 `backend/app/services/resource_evaluation_service.py`

建议新增或调整以下能力：

- `evaluate_resources(resources, user_context)`：主入口保留
- `_split_cached_and_uncached(resources)`：拆分缓存命中与未命中
- `_build_batches(resources, token_budget)`：按预算组批
- `_evaluate_batch(batch, user_context)`：批量调用节点
- `_merge_results(all_resources, cached_map, batch_results)`：恢复顺序并补齐结果

### 关键设计点

- 缓存粒度仍按单资源 URL 维持不变
- 批量只是调用策略变化，不改变缓存键模型
- 批次并发度应可配置，避免瞬时并发过高

## 8.2 `backend/app/agents/nodes/resource_evaluator.py`

建议改造点：

- `evaluate()` 从单资源改为多资源
- `fallback_data = LLMFallback.resource_evaluation(resources, user_context)`
- `generate_json()` 返回后按列表归一化
- `normalize` 从单条版本升级为批量版本

建议拆分方法：

- `_normalize_single_resource(input_resource, evaluated_resource)`
- `_normalize_batch(input_resources, evaluated_resources)`
- `_match_evaluated_resources(input_resources, evaluated_resources)`

## 8.3 `backend/app/utils/llm_fallback.py`

现有 `resource_evaluation()` 已接收列表输入，但当前只在单资源模式下被调用。

改造后应确认：

- fallback 输出数量与输入数量一致
- 输出顺序与输入一致
- 对缺失字段做稳定补齐

## 8.4 `backend/app/agents/prompts/resource_evaluator.md`

建议改造内容：

- 删除不必要的长示例
- 缩短字段说明
- 明确批量输出一致性规则
- 约束 `summary` 和 `reason` 长度，避免输出膨胀

建议长度控制：

- `summary`：40 ~ 80 字
- `reason`：30 ~ 60 字

这样既能保留可读性，也能控制输出 token。

## 9. 动态分批策略

## 9.1 基本原则

分批目标不是追求“批次越大越好”，而是：

- 尽可能减少固定 prompt 重复
- 同时尽量不跨价格档位

## 9.2 推荐策略

建议采用以下预算：

- `target_input_budget = 28000`
- `max_batch_size = 40`
- `min_batch_size = 10`

分批算法可采用贪心：

1. 从待评估资源列表头部开始
2. 估算加入下一条资源后的批次 token
3. 若未超过预算，则继续加入
4. 若超过预算，则关闭当前批次并开始新批次

估算方式可先用轻量近似值：

```text
estimated_tokens = fixed_prompt_cost + sum(len(serialized_resource) * token_ratio)
```

初期不必追求绝对精确，能保证批次规模稳定即可。

## 9.3 推荐参数

第一阶段建议参数：

- 预算：28k
- 默认 batch size 上限：30
- 批次并发数：2 ~ 4

当观察到实际资源体积更小、输出稳定后，再逐步把上限调整到 40。

## 10. 风险与应对

## 10.1 风险：模型遗漏部分资源

表现：

- 输入 20 条，输出 18 条

应对：

- prompt 强约束长度一致
- 归一化阶段做数量校验
- 对缺失项使用 fallback 或默认值补齐

## 10.2 风险：输出顺序错位

表现：

- `summary` 和 `reason` 对到了错误资源

应对：

- 强制保留原始 `url`
- 优先用 `url` 做映射
- 顺序仅作为最后回退手段

## 10.3 风险：批次过大导致进入高价档

表现：

- 虽然总 token 下降，但单次单价升高

应对：

- 设置 28k 软预算
- 记录每批估算 token 和实际批大小
- 通过配置动态下调 batch size

## 10.4 风险：单次失败影响范围扩大

表现：

- 一个批次失败，影响多条资源

应对：

- 批量失败时允许拆小重试
- 重试仍失败则逐条 fallback
- 保留现有默认评估降级逻辑

- `resource_eval_total_resources`

- `resource_eval_timeout_count`

- 平均每任务资源评估耗时
- 每任务总成本
- 资源评估质量是否下降

## 12. 灰度上线建议

建议按三阶段推进：

### 阶段 1：先做 prompt 瘦身

目标：

- 降低固定开销
- 为批量化打基础

动作：

- 精简 prompt
- 缩短输出字段长度要求

### 阶段 2：引入可配置 micro-batch

目标：

- 在低风险前提下验证真实收益

动作：

- 增加批量评估能力
- 默认 `batch_size = 10` 或基于 28k 动态预算
- 保留开关，支持快速回退到逐条模式

### 阶段 3：按观测结果放大批次

目标：

- 最大化降低请求数和 token

动作：

- 将上限从 10 调整到 20、30、40
- 同步观察成本、质量、失败率

---

## 13. 预期收益

在当前“固定 prompt 较重、单条资源较轻”的前提下，改造后的主要收益包括：

- LLM 请求数显著下降
- 输入 token 明显下降
- 单任务资源评估成本下降
- 评估阶段整体耗时下降
- 日志与超时噪音降低

按此前粗估，若资源规模在 20 ~ 100 条之间，采用合理 micro-batch 后，资源评估阶段输入成本通常有机会下降 **70% ~ 85%**。实际收益取决于：

- 每条资源的真实文本长度
- 用户上下文长度
- prompt 是否同步瘦身
- batch size 是否稳定控制在低价格档位内

---

## 14. 最终建议

综合当前代码现状、价格分档和改造收益，建议采用以下方案：

1. 先精简 `backend/app/agents/prompts/resource_evaluator.md`
2. 将资源评估改造成“缓存优先 + cache miss 分批评估”
3. 批量模式采用动态 micro-batch，而非全量一批
4. 默认目标控制在单批 `28k` 输入 token 以内
5. 初始批次上限建议设为 `20 ~ 30`
6. 保留逐条模式作为兜底开关，便于灰度和回退

一句话总结：

**当前资源评估本质上是伪批量；最优优化路径不是“大批量一次梭哈”，而是“prompt 瘦身 + 动态 micro-batch + 单资源缓存复用”。**
