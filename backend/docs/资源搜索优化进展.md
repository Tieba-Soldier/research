# 当前资源搜索优化进展与待优化点

## 1. 文档目的

这份文档用于说明当前项目中“资源搜索”这一段的优化进展，明确：

1. 已经完成了哪些优化；
2. 这些优化目前验证到了什么程度；
3. 还存在哪些待优化点和风险；
4. 下一步建议优先做什么。

本文只聚焦“资源搜索”链路，不展开整个推荐系统的其他模块。

---

## 2. 当前结论

当前资源搜索优化已经完成了大半，状态可以概括为：

- **架构优化已基本落地**
- **国内搜索能力已经接入**
- **主链路已经切到新的搜索实现**
- **搜索可用性已经比早期版本明显提升**
- **但搜索优化还没有完全收口，仍有几个关键问题会影响稳定性和结果一致性**

可以用一句话总结：

> 资源搜索已经从“单一 Tavily 直连”升级到了“多 Provider + 缓存 + fallback + 中文导向 query”的版本，但测试和缓存一致性还需要再补一轮。

---

## 3. 当前搜索链路现状

### 3.1 主链路已经切到 v2

当前 API 路由已经使用新的推荐服务：

- `backend/app/api/routes/recommendation.py`
- `backend/app/services/recommendation_service_v2.py`

而 `recommendation_service_v2.py` 中的搜索阶段已经统一走：

- `backend/app/services/search_service.py`

这说明线上主链路已经不再依赖旧版 `ResourceSearchService`。

### 3.2 搜索不再是单一 Tavily 直连

当前搜索层已经有统一 provider 工厂：

- `backend/app/tools/search_provider.py`
- `backend/app/tools/search_provider_factory.py`
- `backend/app/tools/tavily_tool.py`
- `backend/app/tools/bocha_tool.py`

也就是说，搜索能力已经从原来的“写死 Tavily”升级成了“按配置选择 provider，并支持 fallback”。

### 3.3 已支持国内搜索导向

当前配置层已经有统一搜索配置：

- `SEARCH_PROVIDER`
- `SEARCH_REGION`
- `SEARCH_FALLBACK_PROVIDER`
- `ENABLE_BOCHA`
- `ENABLE_TAVILY`

并且搜索 query 生成逻辑已经开始支持中文导向扩展，适配：

- 教程类 query
- 中文文档类 query
- 实战类 query

这说明“国内资源优先”的优化方向已经真正开始落地，而不只是停留在文档阶段。

## 4. 已完成的优化项

## 4.1 搜索 Provider 抽象化

已完成内容：

- 定义统一搜索结果结构 `SearchResult`
- 定义统一 provider 协议
- 接入 `TavilyTool`
- 接入 `BochaTool`
- 提供 `search_provider_factory`

优化价值：

- 搜索能力不再和 Tavily 强耦合
- 可以按环境切换搜索源
- 具备继续扩展其他国内搜索源的基础

当前判断：**已完成**

---

## 4.2 搜索 fallback 机制

已完成内容：

- 搜索主 provider 失败时，自动尝试 fallback provider
- fallback provider 为延迟初始化，不会在启动阶段全部创建

优化价值：

- 某个搜索 API 波动时，推荐流程不至于直接全挂
- 为国内优先、全球兜底提供了基础能力

当前判断：**已完成基础版本**

说明：

fallback 逻辑已经存在，但缓存一致性还没完全处理好，见后文“待优化点”。

---

## 4.3 搜索缓存按 provider 和 region 隔离

已完成内容：

- 搜索缓存 key 已从单纯 `query` 扩展为：
  - `provider`
  - `region`
  - `query`

位置：

- `backend/app/services/cache_service.py`

优化价值：

- 避免不同搜索源之间的结果缓存互相覆盖
- 为 `global / cn / hybrid` 搜索模式提供缓存隔离

当前判断：**基础设计已完成**

说明：

缓存 key 设计本身是对的，但在 fallback 命中后的写缓存行为上还有一个一致性问题，后文单独说明。

---

## 4.4 多 topic 并发搜索

已完成内容：

- 搜索服务已经对多个 topic 使用 `asyncio.gather(...)` 并发搜索

位置：

- `backend/app/services/search_service.py`

优化价值：

- 减少多个 topic 逐个串行搜索带来的总时延
- 资源搜索阶段已经不是单线程串行慢跑

当前判断：**已完成**

---

## 4.5 中文资源导向 query 扩展

已完成内容：

- 在 `SEARCH_REGION in ["cn", "hybrid"]` 时，自动扩展中文导向 query
- 当前已包含：
  - `教程`
  - `中文文档`
  - `实战`

位置：

- `backend/app/services/search_service.py`

优化价值：

- 提高中文学习资源、国内教程、中文文档的命中率
- 为国内搜索体验打下基础

当前判断：**已完成第一版**

---

## 4.6 国内站点资源类型识别增强

已完成内容：

- 已将部分国内站点纳入资源类型识别规则
- 当前已覆盖：
  - `bilibili.com`
  - `gitee.com`
  - `juejin.cn`
  - `csdn.net`
  - `zhihu.com`
  - 腾讯云 / 阿里云 / 华为云 / 火山引擎等站点

优化价值：

- 资源类型识别比过去更贴近中文开发者生态
- 有利于后续按资源类型排序或加权

当前判断：**已完成第一版**

## 6. 当前待优化点

下面这些是当前最值得继续收尾的点。

## 6.1 fallback 结果写入缓存时仍按主 provider 记账

这是当前最关键的搜索层一致性问题。

现状：

- 读缓存时使用的是 `primary_provider`
- 搜索失败后可能实际走了 `fallback_provider`
- 但写缓存时仍然使用 `primary_provider` 作为缓存命名空间

结果：

- fallback provider 搜到的结果，会被写进主 provider 的缓存空间
- 主 provider 恢复后，仍可能继续读到 fallback 的历史结果

影响：

- provider 级缓存隔离被破坏
- 结果来源不准确
- 调试时很难判断当前结果到底来自谁

优先级：**高**

建议：

- `_search_with_fallback()` 不只返回 `results`
- 还应返回“实际命中的 provider 名称”
- 写缓存时使用实际 provider，而不是主 provider

---

## 6.2 旧版 `ResourceSearchService` 仍然保留，且仍写死 Tavily

当前旧文件仍在：

- `backend/app/services/resource_search_service.py`

它的问题是：

- 仍直接依赖 `TavilyTool`
- 没接入 provider 工厂
- 没接入新的搜索策略

虽然主 API 已经切到 v2，不一定还在用这条链路，但它继续留在仓库里会带来几个风险：

1. 后续开发容易误用旧逻辑；
2. 维护者阅读代码时会误判当前真实搜索实现；
3. 新旧逻辑长期并存会让后续改造成本继续上升。

优先级：**中高**

建议：

- 如果确认已废弃，删除或标注 deprecated
- 如果还要保留，至少改成与新 provider 体系一致

---

## 6.3 中文导向 query 实际只生效了前 3 个

当前 `search_service.py` 里虽然生成了更多中文导向 query，包括：

- `B站`
- `掘金`

但最终返回时直接截成了前 3 个。

结果：

- 代码“看起来”支持平台导向
- 但实际执行里，`B站` 和 `掘金` 导向查询并没有被用到

影响：

- 国内资源增强的效果低于代码表面上看起来的程度

优先级：**中**

建议：

- 重新设计 query 优先级
- 不要简单裁切前 3 个
- 可以按主题类型动态选择：
  - 文档型主题优先“中文文档”
  - 入门型主题优先“教程”
  - 实战型主题优先“实战”
  - 偏实践型主题再加入“B站”或“掘金”

---

## 6.4 监控指标仍然残留 `tavily_calls`

当前推荐服务的 metrics 日志仍在使用：

- `tavily_calls`

但现在搜索层已经不是纯 Tavily 了。

这会导致：

- 指标语义不准确
- 日志看起来像搜索仍然只依赖 Tavily
- 后续排障容易误导

优先级：**中**

建议：

- 改成更通用的：
  - `search_provider_calls`
  - `primary_provider`
  - `fallback_triggered`

---

## 6.5 缺少“搜索优化是否生效”的专项测试

当前 E2E 更像是在验证：

- 整个推荐流程能不能跑完

但还没有真正验证这些搜索优化本身：

1. 第二轮请求是否命中搜索缓存；
2. fallback 是否真的生效；
3. provider 切换后是否返回预期结果；
4. `cn / global / hybrid` 三种模式是否行为正确；
5. 中文导向 query 是否比普通 query 带来更好的中文资源命中率。

优先级：**高**

建议：

- 增加搜索层专项测试
- 至少补 4 类用例：
  1. cache hit
  2. fallback hit
  3. provider switch
  4. region mode behavior

---

## 6.6 当前缺少“搜索阶段独立耗时”的稳定证据

虽然文档和代码里都已经加入了 metrics 日志位，但目前仓库里没有很清晰、可复查的搜索阶段性能产物。

当前可见的是：

- 总链路 E2E 耗时

但还缺：

- 纯搜索阶段耗时
- 每个 topic 的查询数
- 缓存命中率
- fallback 触发次数
- provider 成功率

优先级：**中**

建议：

- 统一保留一份搜索性能报告
- 至少包含：
  - `search_duration`
  - `queries_per_topic`
  - `cache_hit_rate`
  - `fallback_count`
  - `resources_before_dedup`
  - `resources_after_dedup`

3. 清理或废弃旧版 `ResourceSearchService`

### P2

5. 把 `tavily_calls` 指标改成通用搜索指标
6. 补一份独立的搜索性能报告，区分“整链路耗时”和“搜索阶段耗时”

## 9. 最终总结

当前资源搜索优化已经完成了以下关键转变：

- 从单一 Tavily 直连，升级到多 Provider 搜索架构；
- 从无国内搜索支持，升级到可接入国内搜索源；
- 从单纯 query 搜索，升级到支持中文导向 query；
- 从简单缓存，升级到包含 provider/region 维度的缓存设计；
- 从旧版串行、耦合较重的搜索逻辑，升级到新的统一搜索服务。

但当前还存在几个必须继续处理的问题：

- fallback 结果缓存命名不准确；
- 旧搜索实现仍留在仓库中；
- 中文搜索增强还没有完全发挥出来；
- 搜索专项验证还不充分；
- 搜索指标和验证产物还不够统一。

所以更准确的表述不是“资源搜索已经优化完成”，而是：

> 资源搜索优化已经完成主体重构，当前正处于从“功能已落地”走向“效果已验证、边角已收口”的阶段。
