# 项目优化建议与路线图

日期：2026-04-19

## 1. 评估范围

本次评估基于当前仓库中的前后端实现与已有诊断文档，重点查看了以下模块：

- 后端主链路：`backend/app/services/recommendation_service_v2.py`
- 推荐接口：`backend/app/api/routes/recommendation.py`
- 资源接口：`backend/app/api/routes/resource.py`
- 搜索、评估、学习路径、练习题服务：`backend/app/services/*.py`
- Agent 节点与 LLM 调用：`backend/app/agents/nodes/*.py`、`backend/app/tools/llm_client.py`
- 前端页面与组件：`frontend/src/pages/RecommendationPage.vue`、`frontend/src/components/*.vue`
- 数据模型与索引：`backend/app/models/*.py`、`backend/alembic/versions/002_add_indexes.py`

结论先说：项目当前已经具备“可跑通资源推荐链路”的基础能力，服务拆分、缓存、兜底和结果持久化都比初版成熟不少；但如果希望它从“可演示 MVP”走向“可稳定扩展的产品”，仍然需要在架构边界、功能闭环、性能控制和工程治理四个层面继续优化。

## 2. 当前做得比较好的地方

### 2.1 服务拆分已经初步成型

`TopicService`、`SearchService`、`ResourceEvaluationService`、`LearningPathService`、`PracticeTaskService` 已经从单体推荐服务中拆开，主流程集中在 `RecommendationService` 编排，结构比“一坨业务全塞一个文件”清晰很多。

### 2.2 外部依赖失败时有基本兜底

资源评估、学习路径、练习题生成都提供了 fallback，说明系统已经考虑到 LLM 和外部搜索服务不稳定的问题，这一点对 Agent 类应用很重要。

### 2.3 已经开始关注性能

当前实现里已经有三类性能意识：

- 主题并发搜索
- 资源并发评估
- Redis 缓存与数据库索引

这说明项目已经不只是“先做出来”，而是在往“跑得动、跑得快”靠拢。

### 2.4 结果数据有持久化基础

主题、资源、学习路径、练习任务都已落库，后续做历史查询、推荐复用、用户行为学习，都有继续扩展的空间。

## 3. 核心优化建议

## 3.1 架构层

### A. 把“进程内后台任务”升级为“独立任务执行体系”

当前 `POST /api/recommendations` 仍通过 FastAPI `BackgroundTasks` 在 Web 进程内执行推荐任务。这个方案适合 MVP，但有几个明显问题：

- Web 服务重启时任务会中断
- 无法横向扩容 worker
- 缺少任务重试、超时回收、并发配额控制
- 长任务会和 API 请求共享同一进程资源

建议：

- 短期：引入任务表里的 `started_at / finished_at / retry_count / worker_id`
- 中期：迁移到 Celery / Dramatiq / RQ / Arq 这类独立 worker 架构
- 同时为任务增加“排队中、执行中、部分完成、失败可重试”状态

预期收益：系统稳定性和可运维性会明显提升，是从 Demo 到生产化最关键的一步。

### B. 补齐“请求配置”在后端的真实生效链路

前端已经暴露了：

- `include_video`
- `include_articles`
- `include_official_docs`
- `include_github`
- `include_practice_tasks`
- `max_topics`
- `max_resources_per_topic`

但当前后端创建任务时基本只使用了 `user_input`，这些配置没有被真正持久化并传入后续服务。结果就是：

- 用户看到了配置项
- 但推荐结果并不会按这些配置真实变化

建议：

- 在 `recommendation_tasks` 增加配置字段，或新增 `recommendation_task_options` 表
- 所有服务改为消费“任务级配置”而不是仅读全局 `settings`
- 搜索阶段根据配置过滤资源类型
- 如果关闭 `include_practice_tasks`，应跳过练习题生成阶段

预期收益：把“看起来有的功能”变成“真正生效的功能”，这是当前最值得优先补齐的功能性缺口。

### C. 重构“主题-资源”关系模型，避免去重后丢失关联

当前 `resources` 表只有一个 `topic_id`，而 `SearchService` 会对所有主题的搜索结果按 URL 去重。这样会带来一个结构性问题：

- 同一个资源可能同时适合多个主题
- 但去重后只会保留一个 `topic_id`
- 后续学习路径、资源解释、主题视图都会丢失多对多关系

建议：

- 保留 `resources` 作为资源主表
- 新增 `topic_resources` 关联表，字段至少包括：`topic_id`、`resource_id`、`raw_score`、`evaluation_score`、`rank`
- 去重时保留资源主记录，但主题关联通过中间表保存

预期收益：推荐结果的解释性、准确性和后续扩展性都会更好，尤其适合做“按主题查看资源”与“学习路径按阶段关联资源”。

### D. 收敛 LLM / Search 客户端构造方式

现在多个节点各自 `LLMClient()`，搜索服务也自行 new `TavilyTool()`。这样的问题是：

- 连接复用不稳定
- provider/model/timeout 配置散落
- 难以统一做监控、限流、熔断、降级

建议：

- 做一个统一的 provider factory
- 客户端实例在应用启动时初始化，按 provider 复用
- 把 model、timeout、temperature、max_tokens 等纳入配置中心
- 对每类外部调用打统一埋点

预期收益：配置治理更清晰，后续切模型或增加供应商成本更低。

## 3.2 功能层

### A. 增加“结果可解释 + 用户反馈闭环”

当前系统能给出资源、路径、练习题，但用户反馈没有反哺推荐链路。下一阶段最值得补的不是更多按钮，而是闭环：

- 用户是否点开资源
- 是否标记“有帮助/没帮助”
- 是否已学完
- 是否希望“更基础/更深入/更多实战”

建议：

- 为资源增加 like/dislike、not_relevant、regenerate_similar
- 为学习路径增加“太简单/太难/顺序不合理”反馈
- 用这些反馈驱动二次排序与 prompt 调整

预期收益：系统会从“一次性推荐器”走向“可逐步个性化的学习助手”。

### B. 做“任务历史与结果复访”

当前用户体验更像单次生成。若刷新页面或过几天回来，缺少完整的历史入口和结果复访入口。

建议：

- 增加任务列表页 / 最近任务面板
- 支持按状态筛选：进行中、已完成、失败
- 支持查看上次输入、主题、资源、路径、练习题
- 支持复制旧任务重新生成

预期收益：用户留存和可用性提升明显，也更符合“研究/学习型工具”的使用方式。

### C. 资源操作应真正进入“用户体系”

`mark-studied` 和 `favorite` 目前仍是固定 `user_id = 1` 的写法，说明功能已存在，但还没真正产品化。

建议：

- 接入认证体系
- 所有用户行为数据与真实用户绑定
- 返回资源列表时回填 `studied` / `favorite` 状态
- 支持按“未学习/已收藏”过滤结果

预期收益：用户行为数据才能成为后续个性化推荐的基础。

### D. 让推荐结果支持“继续研究”

现在结果主要是展示。可以进一步增加二次操作：

- 只看官方文档
- 只看视频
- 只看高分资源
- 生成该主题的进阶问题
- 基于已学资源重新生成下一阶段路径

这类能力会让产品从“推荐页”升级成“学习工作台”。

## 3.3 性能层

### A. 搜索阶段建议从“跨主题并发”升级为“有上限的分层并发”

当前并发主要发生在“多个主题一起搜”，但单个主题内部 query 仍是串行的。这样在主题较少、每主题 query 较多时，性能并不理想。

建议：

- 在 query 级别也做并发
- 使用 `asyncio.Semaphore` 控制总并发数
- 为 Tavily 增加每秒请求数限制，避免把外部 API 打爆

预期收益：延迟进一步下降，同时不会因为完全放开并发导致超限。

### B. 资源评估不建议“一条资源一次 LLM”

现在资源评估基本是逐条发起 LLM 调用。随着资源数增加，这会很快变成延迟和成本的大头。

建议：

- 先用 Tavily score + 规则做第一轮粗排
- 只对 Top N 做 LLM 精评
- 或者按批次让 LLM 一次评估多条资源

预期收益：延迟、Token 成本和失败面都会显著下降。

### C. 缓存键需要更精细，避免误命中

当前资源评估缓存主要按 `url` 缓存，但资源评估实际上依赖 `user_context`。这会带来两个风险：

- 同一 URL 在不同主题下拿到相同推荐理由
- 个性化上下文被缓存“污染”

建议：

- 缓存键改为 `provider + model + normalized_topic + url + version`
- 为 prompt 变更引入 cache version

预期收益：缓存命中仍在，但不会牺牲推荐语义准确性。

### D. 降低数据库提交频率

当前主链路里状态更新、主题保存、资源保存、结果保存分散在多个 commit 中。对稳定性有好处，但会增加数据库交互次数。

建议：

- 任务状态更新保留分阶段提交
- 主题、资源、练习题保存尽量批量化
- 对纯新增数据优先使用 bulk insert / returning
- 明确哪些步骤需要强一致，哪些步骤接受最终一致

预期收益：数据库压力更低，长任务吞吐更好。

### E. 前端轮询可以升级为 SSE / WebSocket

当前前端每 2 秒轮询一次任务状态。这个方式简单，但成本并不低：

- 空轮询很多
- 多用户并发时对后端压力大
- 用户看到的进度也不够实时

建议：

- 短期改成“指数退避轮询”
- 中期升级为 SSE
- 若后续需要实时日志流，再考虑 WebSocket

预期收益：更省资源，也更适合长任务。

## 3.4 工程治理层

### A. 建议建立三层测试体系，而不只依赖 E2E

当前已有 E2E 脚本是加分项，但还不够。推荐系统里最脆弱的是“数据契约”和“降级路径”，这些更适合单测和契约测试兜底。

建议测试拆成三层：

- 单元测试：topic normalize、resource dedupe、score 计算、fallback 生成
- 契约测试：LLM 输出解析、schema 校验、路由响应结构
- E2E 测试：真实外部服务场景

预期收益：一旦未来再重构 prompt、节点、缓存，不容易把主链路悄悄改坏。

### B. 增加可观测性，不只看日志

现在日志已经有一些 metrics 字段，但还不够系统化。建议补齐：

- 每阶段耗时
- 每任务 LLM 调用数
- 每任务搜索调用数
- 缓存命中率
- 降级触发率
- 失败原因分布

建议接 Prometheus + Grafana，或者至少统一 structured logging。

### C. 收紧 API 与安全配置

当前存在一些明显的“开发态配置”：

- `allow_origins=["*"]`
- `allow_credentials=True`
- `create_async_engine(..., echo=True)`

建议：

- 按环境切换 CORS
- 生产环境关闭 SQL echo
- 对推荐接口增加限流与请求大小限制

### D. API 类型约束可以更严格

虽然定义了不少 schema，但路由层仍大量使用 `response_model=dict`。这会让接口文档和返回结构的约束变松。

建议：

- 定义统一 `ApiResponse[T]`
- 路由返回强类型对象
- 前后端共享或同步维护接口契约

预期收益：后续改接口时更容易第一时间发现破坏性变更。

## 4. 优先级建议

### P0：建议立刻处理

1. 后端真正消费前端传入的推荐配置
2. 把后台任务从进程内执行改造成可恢复的任务体系
3. 修正资源评估缓存键，避免上下文污染
4. 把资源操作接入真实用户体系

### P1：建议下一阶段处理

1. 重构主题-资源多对多模型
2. 为推荐结果增加历史记录和复访能力
3. 降低 LLM 逐条评估开销
4. 轮询改为 SSE 或至少退避轮询
5. 增加测试体系与监控指标

### P2：可以作为产品增强项

1. 用户反馈驱动的二次推荐
2. 资源筛选、继续研究、再次生成
3. 多供应商智能路由
4. 个性化学习路径与进度跟踪

## 5. 一个可执行的三阶段路线图

### 第一阶段（1 周）

- 打通任务配置真实生效
- 收紧缓存键
- 关闭生产无关配置
- 补关键单测

目标：让现有功能“说到做到”。

### 第二阶段（2-3 周）

- 引入独立 worker
- 增加任务历史
- 改造轮询机制
- 补监控指标

目标：让系统“稳定可持续运行”。

### 第三阶段（1-2 个月）

- 重构主题-资源关系
- 接入用户反馈闭环
- 优化 LLM 评估策略
- 做个性化推荐与学习进度

目标：让产品从“推荐工具”进化为“学习助手”。

## 6. 总结

如果只看“有没有把资源推荐功能做出来”，这个项目已经达标；如果看“能不能继续迭代成一个稳定、可信、可扩展的产品”，当前最值得发力的不是再堆新功能，而是把以下三件事做扎实：

1. 任务执行体系独立化
2. 请求配置与数据模型闭环化
3. 性能与工程治理体系化

我对这个项目的判断是：基础不错，方向也对，已经过了“从 0 到 1”的门槛；接下来真正拉开差距的，会是架构治理和产品闭环，而不是再多接一个模型或再多加几个按钮。
