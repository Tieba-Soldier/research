# 资源推荐功能故障诊断

日期：2026-04-19

## 结论摘要

当前“根据用户输入推荐相应资源”的失败，不是单一问题，而是三类问题叠加：

1. 现网/旧进程层面：历史回归测试 5 个用例里只有 1 个被判定为通过，但这个“通过”结果仍然包含大量空字段，说明结果质量并不达标。
2. 当前源码层面：`recommendation_service_v2` 这一条链路里存在多处数据契约不一致，当前代码即使启动成功，也很容易在“资源评估 / 学习路径 / 练习任务”阶段出错。
3. 配置层面：当前源码在本地无法直接启动，因为 `.env` 中存在 `DASHSCOPE_MODEL`，但 `Settings` 没有声明该字段，启动时会直接抛 `ValidationError`。

换句话说，现有测试里“只有 1/5 通过”只是表面现象；真正的问题是推荐链路本身已经有结构性断裂。

## 本次分析方式

我做了两类核查：

1. 读取项目中的已有回归测试和测试产物：
   - `backend/tests/e2e_test.py`
   - `backend/test_report.txt`
   - `backend/test_result.json`
   - `backend/test_output.log`
   - `backend/test_output_dashscope.log`
2. 直接检查当前后端源码中推荐主链路的实现：
   - `backend/app/api/routes/recommendation.py`
   - `backend/app/services/recommendation_service_v2.py`
   - `backend/app/services/topic_service.py`
   - `backend/app/services/resource_evaluation_service.py`
   - `backend/app/services/learning_path_service.py`
   - `backend/app/services/practice_task_service.py`
   - `backend/app/agents/nodes/*.py`
   - `backend/app/utils/llm_fallback.py`
   - `backend/app/core/config.py`

另外我尝试用当前源码在 `127.0.0.1:8001` 拉起服务，结果启动阶段即失败，错误是：

`ValidationError: DASHSCOPE_MODEL Extra inputs are not permitted`

这说明磁盘上的当前源码和 `localhost:8000` 正在运行的那个进程很可能不是同一版状态。

## 已有测试结果

项目内现有报告显示：

- 总测试数：5
- 通过：1
- 失败：4
- 通过率：20%

失败类型主要有三种：

1. `list index out of range`
2. `LLM generation failed: SiliconFlow API error: Request timed out`
3. `Failed to parse JSON after retry`

更关键的是，唯一的“通过”样本仍然包含空字段：

- `resource.title`
- `resource.url`
- `resource.summary`
- `resource.reason`
- `topic.normalized_topic`
- `practice_task.task_text`

也就是说，测试脚本当前把“数量不为 0”当成了通过，但没有把“关键字段为空”算作失败。

## 核心问题清单

### P0：当前源码无法直接启动

证据：

- `backend/app/core/config.py:5-46` 定义了 `Settings`
- `backend/.env:18` 存在 `DASHSCOPE_MODEL=qwen-max`
- 但 `Settings` 中没有 `DASHSCOPE_MODEL` 字段

影响：

- 当前源码在导入 `settings = Settings()` 时就会失败，服务根本起不来。
- 因此你现在看到的 `localhost:8000` 很可能是之前启动的旧进程，而不是当前磁盘代码。

### P0：练习任务链路的数据结构已经断裂

证据：

- `backend/app/services/recommendation_service_v2.py:187-196`
  - 这里把 `practice_tasks` 当成“任务列表”使用
- `backend/app/services/recommendation_service_v2.py:324-343`
  - `_save_practice_tasks()` 里直接 `for practice_task in practice_tasks`
  - 然后调用 `practice_task.get(...)`
- `backend/app/services/practice_task_service.py:31-37`
  - `generate_practice_tasks()` 直接返回 `self.generator.generate(topics)` 的结果
- `backend/app/agents/nodes/practice_task_generator.py:21-37`
  - `generate()` 返回的是一个字典，形如 `{"tasks": [...]}`

影响：

- `recommendation_service_v2` 期待的是 `List[Dict]`
- `PracticeTaskGenerator` 实际返回的是 `Dict[str, Any]`
- 一旦走到 `_save_practice_tasks()`，就会按字典迭代，迭代出来的是键名而不是任务对象，后续 `.get(...)` 会出错

这属于必修复项，因为它会直接破坏推荐流程后半段。

### P0：资源评估链路的函数签名完全对不上

证据：

- `backend/app/services/resource_evaluation_service.py:80-84`
  - 调用：`self.evaluator.evaluate(resource, user_context)`
  - 这里传入的是“单个资源 dict + 用户上下文字符串”
- `backend/app/agents/nodes/resource_evaluator.py:21-30`
  - 定义：`evaluate(self, tavily_results: List[Dict], topic: str, task_id: int = None)`
  - 这里期待的是“资源列表 + topic 字符串”
- `backend/app/services/resource_evaluation_service.py:83`
  - 调用了 `self.evaluator.calculate_final_score(evaluated)`
- 但项目里没有 `calculate_final_score` 这个方法定义

影响：

- 评估器入参和调用方式不一致
- 即使 LLM 返回成功，这里也会因为签名/方法缺失而回退或报错
- 资源评分阶段会非常不稳定，结果经常退化成兜底值，或者直接导致任务失败

### P1：主题字段命名不一致，容易写出空主题

证据：

- `backend/app/agents/nodes/topic_extractor.py:28-36`
  - 提取器返回结构使用的是 `raw_text` / `normalized_topic`
- `backend/app/agents/nodes/topic_normalizer.py:30-41`
  - fallback 又构造了 `original` / `normalized`
- `backend/app/services/recommendation_service_v2.py:250-263`
  - `_save_topics()` 只读取 `original` / `normalized`
- `backend/app/utils/llm_fallback.py:33-41`
  - `topic_extraction()` fallback 返回的还是 `raw_text` / `normalized_topic`

影响：

- 只要走到 extractor fallback，而 normalizer 没把字段正确转成 `original/normalized`
- 最终入库时 `normalized_topic` 就会被写成空字符串
- 这和 `backend/test_result.json` 中出现的空主题字段现象是一致的

### P1：学习路径服务接口和生成器接口不一致

证据：

- `backend/app/services/learning_path_service.py:15-39`
  - `generate_learning_path(topics, resources)` 设计上需要主题和资源
- `backend/app/services/learning_path_service.py:34`
  - 实际调用：`self.generator.generate(topics, resources)`
- `backend/app/agents/nodes/learning_path_generator.py:21-39`
  - 生成器签名却是：`generate(self, topics: List[str], task_id: int = None)`
  - 第二个参数不是 `resources`，而是 `task_id`

影响：

- 资源列表被错误地当成 `task_id` 传进去了
- 学习路径生成器实际上没有消费资源信息
- 即使功能不报错，也生成不了真正“基于资源排序结果”的学习路径

### P1：fallback 的输入类型也不一致

证据：

- `backend/app/agents/nodes/practice_task_generator.py:26`
  - `LLMFallback.practice_tasks(topics[0] if topics else "学习主题")`
- 但在当前 v2 主链路里，`topics` 传入的是 `saved_topics`，它是主题字典列表，不是字符串列表
- `backend/app/agents/nodes/learning_path_generator.py:28`
  - `LLMFallback.learning_path(topics)` 也假设 `topics` 是字符串列表

影响：

- 一旦进入 fallback，提示词和兜底数据里会混入字典对象
- 这会继续扩大格式异常和空字段问题

### P2：测试脚本的通过标准过宽，掩盖了真实质量问题

证据：

- `backend/tests/e2e_test.py:109-138`
  - 脚本会检测空字段
  - 但只要主题数、资源数、练习任务数不为 0，且有学习路径，就把测试判定为 `success = True`
  - `empty_fields` 不会导致失败

影响：

- 测试统计里的“1 个通过”并不代表功能可用
- 它只代表“至少返回了一点东西”
- 这会误导后续排查，让人以为模型偶发失败，而不是主链路本身已经坏了

## 为什么现在会出现“5 个问题几乎只有 1 个通过”

结合代码和测试产物，原因可以概括成一句话：

`外部 LLM/JSON 不稳定 + 内部数据结构不一致 + 测试判定过宽`

具体展开是：

1. 有些请求死在 LLM 超时或 JSON 解析阶段。
2. 即使没死在 LLM 阶段，后面的资源评估、学习路径、练习任务链路也可能因为契约不一致继续出错。
3. 即使侥幸走完流程，返回结果里仍然可能是空标题、空主题、空练习题。
4. 测试脚本没有把“空字段”视为失败，所以把低质量结果误记成了通过。

## 建议的修复顺序

建议按下面顺序修：

1. 先修启动问题
   - 让 `config.py` 接受 `.env` 里的 `DASHSCOPE_MODEL`，或者显式忽略额外字段
   - 否则当前源码无法稳定复现和验证
2. 再统一推荐主链路的数据契约
   - 统一 topic 的字段名
   - 统一 practice task 的返回结构
   - 统一 learning path 的参数结构
   - 统一 resource evaluator 的输入/输出结构
3. 最后再处理 LLM 超时和 JSON 解析鲁棒性
   - 因为如果内部契约不通，光优化模型调用也救不了链路
4. 收紧测试标准
   - 关键字段为空时应直接判失败
   - 否则回归测试没有约束力

## 我对当前状态的判断

如果只问“为什么现在不好用”，答案不是“模型偶尔抽风”，而是：

- 当前推荐主流程同时存在配置问题、接口契约问题和结果校验问题；
- 就算模型表现稳定，当前源码也还没有到可依赖的程度；
- 在修完主链路契约之前，继续扩大测试样本意义不大，因为很多失败不是偶发，而是确定性的实现问题。

## 可直接作为后续修复任务的事项

1. 统一 `topic` 结构：只保留一套字段命名。
2. 统一 `resource_evaluator` 的输入输出：明确是“单资源评估”还是“批量评估”。
3. 统一 `practice_task_generator` 的返回值：服务层和持久化层必须都按同一结构处理。
4. 修正 `learning_path_service -> learning_path_generator` 的参数签名。
5. 修复 `Settings` 与 `.env` 的不兼容。
6. 让 E2E 测试把空字段算作失败。
